# Claude スキル化の分析と設計書

作成日: 2026-07-13

## 0. 分析の前提(何を見たか)

claude.ai のチャット履歴そのものは、この実行環境(リポジトリごとの独立コンテナ)からは参照できない。そのため本分析は、やり取りの**成果物側に残った痕跡**から作業パターンを復元した:

- このリポジトリの全コミット履歴(4件)とコード全体
- 今回のセッションでの依頼内容
- アカウント上のリポジトリ一覧(直近の活動: portfolio / poster-map / nisa)

チャット履歴でしか分からないパターン(質問の仕方、やり直しの頻度、説明の好みなど)は反映できていない。**「このやり取りをよくする」と感じた実際の会話があれば、それを貼ってもらえれば設計を精緻化できる。**

## 1. 復元できた作業パターン

コミット履歴から、このプロジェクトの典型的な1サイクルは次のように読み取れる:

| コミット | 日付 | 読み取れる作業 |
|---|---|---|
| 初回リリース (240件) | 6/30 | Excel/CSV の住所リスト → ジオコーディング → data.js 生成 → 地図アプリ実装 |
| スマホ: 詳細パネルをスライド表示 | 7/1 | 現地(スマホ)で使ってみて気づいた UI 改善 |
| 全表示/配分表示の2モード追加 | 7/3 | Excel のセル背景色で管理している担当配分をアプリに反映 |
| No.58, 59 の座標修正 | 7/4 | 更新版 CSV を根拠に個別の座標を手修正 |

つまり **「Excel が正本 → Claude に渡して JS データへ変換 → 地図で目視確認 → 現地利用 → 誤りを個別修正」** というデータパイプラインを、毎回チャットで口頭指示しながら回している。ここが最も再現性高くスキル化できる。

また、リポジトリ一覧(ポートフォリオサイト、NISA 関連、本マップ)から、**「身の回りの実データを小さな Web ツールにする」**のが継続的な活動パターンだと推測できる。

## 3. 未実装スキルの設計書

### 3.1 geocode-check(座標妥当性チェック)

- **課題**: No.58/59 の座標修正のように、ジオコーディング結果の誤りは目視か現地でしか気づけていない。240 件を人力で見るのは非現実的。
- **トリガー**: 「座標がずれてないか確認して」「ジオコーディングを検証して」、または poster-data-update で大量更新した直後。
- **手順**:
  1. data.js の各件について、address を国土地理院ジオコーダ API(`https://msearch.gsi.go.jp/address-search/AddressSearch?q=`)で再ジオコーディングする(無料・キー不要。1件ずつ間隔を空ける)
  2. 登録座標との距離を計算し、しきい値(例: 300m)を超えた id を「要確認リスト」として住所・両座標・距離付きで報告する
  3. 自動では修正しない。ユーザーが Excel 側で確認してから poster-data-update で反映する
- **注意**: ネットワークポリシーで外部 API に出られない環境では実行不可。その場合は「同一地区内での外れ値検出」(住所の町名でグルーピングし、グループ重心から極端に遠い点を報告)にフォールバックする。
- **成果物**: `.claude/skills/geocode-check/SKILL.md` + `scripts/geocode-check.mjs`

### 3.2 mobile-check(スマホ表示確認)

- **課題**: 主用途が現地のスマホ利用なのに、確認は毎回手動。7/1 の修正はリリース後に気づいた例。
- **トリガー**: styles/ または index.html に触る変更をしたとき。「スマホで確認して」。
- **手順**:
  1. Playwright(この環境に Chromium 内蔵)で幅 375×667 と 1280×800 の 2 サイズでページを開く
  2. Google Maps API キーが動かない環境では、`google.maps` をスタブしてレイアウトのみ確認する(ヘッダー・ボタン・詳細パネルの開閉)
  3. 両サイズのスクリーンショットを撮ってユーザーに送る
- **チェック項目**: 横スクロールが発生しない / モード切替ボタンがタップ可能サイズ(44px 以上)/ 詳細パネルが地図を覆い隠しすぎない / 閉じるボタンが機能する
- **成果物**: `.claude/skills/mobile-check/SKILL.md` + スクリーンショット取得スクリプト

### 3.3 release-check(公開前チェックリスト)

- **課題**: 公開リポジトリに Google Maps API キーが生で入っている(index.html)。キー自体は HTTP リファラ制限をかければ運用上問題ないが、**制限がかかっているかはコードから確認できず、チェックの仕組みがない**。
- **トリガー**: 「公開して」「リリースして」、main へのマージ前。
- **チェック項目**:
  1. API キーに HTTP リファラ制限(公開 URL のみ許可)がかかっているかをユーザーに確認(Google Cloud Console での設定はユーザーしかできない)
  2. `node scripts/validate-data.mjs` が警告ゼロ
  3. 表示件数(count-info)とデータ件数の一致
  4. 個人情報・内部メモがデータに紛れ込んでいないか(description 欄の目視サンプリング)
- **成果物**: `.claude/skills/release-check/SKILL.md`(チェックリストのみ、スクリプト不要)

### 3.4 data-to-webtool(Excel → Web ツール化の型)【汎用・抽象】

- **課題**: poster-map で確立したパターン(Excel の実データ → 検証付き JS データ → 静的1ページアプリ → スマホ対応)は、次の題材(例: NISA の資産一覧、名簿、イベント管理)でもゼロから説明し直すことになる。
- **トリガー**: 「この Excel/スプレッドシートを Web で見られるようにしたい」系の依頼。
- **スキルが定める型**:
  1. **正本を決める**: Excel を正本のまま「CSV で渡す→変換スクリプトで JS 生成」とする(生成物を手編集しない)。変換スクリプトは必ずリポジトリに残す
  2. **検証を最初に書く**: データの不変条件(id 連番、必須列、値域)を CLI スクリプトにしてから UI を作る
  3. **構成は3ファイル**: `data.js`(生成物)/ `app.js`(ロジック)/ `style.css`。ビルド工程を持ち込まない
  4. **スマホファースト**: 375px で先にレイアウトを決める
  5. **公開前チェック**: API キー・個人情報の扱いを release-check に準じて確認
- **置き場所**: リポジトリ横断で使うため、プロジェクト内ではなく個人スキル(`~/.claude/skills/`、または各リポジトリにコピー)として持つのがよい。
- **成果物**: `SKILL.md`(手順書)+ 雛形一式(index.html / validate の骨組み)

## 4. スキル以外で効く改善(併せて実施・提案)

1. **CLAUDE.md**(実装済み): 毎セッション冒頭の「このプロジェクトは何か」の説明が不要になる。スキルより先に効く最重要の改善。
2. **元 CSV の扱い**: 現在は data.js だけが正本で、元 CSV はチャット添付で渡す運用。CSV をリポジトリの `data/` に置いて data.js を生成物にすると、差分管理と再生成が確実になる(個人情報の懸念があるなら現運用のままでよい — これは ronten-seiri 向きの論点)。
3. **API キーのリファラ制限**: コード変更ではなく Google Cloud Console 側の設定。公開リポジトリなので、未設定なら早めの対応を推奨。

## 5. 使い方

- このブランチをマージすると、次のセッションから Claude が CLAUDE.md を自動で読み、`.claude/skills/` の3スキルを状況に応じて自動起動する(明示的に `/poster-data-update` のように呼んでもよい)。
- §3 の未実装スキルは、必要になったタイミングで「geocode-check を実装して」と言えば、この設計書に沿って作成できる。
